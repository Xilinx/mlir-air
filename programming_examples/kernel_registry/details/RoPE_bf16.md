# RoPE (BF16, half-split) — Kernel Detail

> Rotary Position Embedding applied to the Q and K projections of a decoder-only
> LLM. BF16 in/out, per-element (no reduction, no non-linearity beyond the
> precomputed cos/sin). **Half-split** convention (HuggingFace Llama
> `rotate_half`), pairing `(x[i], x[i+head_dim/2])` — **not** the interleaved
> even/odd variant.
> Shape is `rows × head_dim`: `input[rows, head_dim]`, `lut[rows, head_dim]` →
> `output[rows, head_dim]`, where `rows = n_heads · seq` (prefill) or `n_heads`
> (decode), and each LUT row holds the position's cos/sin in the concatenated
> layout `[cos_0..cos_{half-1}, sin_0..sin_{half-1}]`.
>
> Companion: [`../supported_kernels.md`](../supported_kernels.md) · [`../README.md`](../README.md)
> **Scope: NPU2 (Strix / AIE2P) only.** Measured on real NPU2, June 2026. Reproduce commands in "How to reproduce" below.

---

## Builder

```
programming_examples/rope_halfsplit/rope_halfsplit.py
  build_module(rows, head_dim, np_dtype, herd_x=8, herd_y=1)
```

Driven by `rope_halfsplit.py`'s CLI; the example also has a `Makefile`. Single
code-generation path: external C++ microkernel `rope_halfsplit.cc → rope.o`
(symbol `@rope`), `link_with`-ed into the herd. This is the **exact kernel
llama-3.2-1B links** — `external_kernels.py:compile_rope` compiles
`rope_halfsplit.cc` to the same `rope.o`. Each `head_dim` row is streamed
L3→L1→L3 (one row per DMA), the kernel rotates it, the result is streamed back;
rows are interleaved across an `herd_x × herd_y` AIE grid (tile `t` handles rows
`t, t+total_tiles, …`). Rows are independent (no cross-row dependency), so
`herd_x` (AIE columns) is the scaling knob.

**Note on llama usage.** The standalone harness rotates the same rows llama does,
linking the same `rope.o`. In the e2e model the RoPE launch is one of six launches
in a fused multi-launch ELF (`rms_gemms_rope_multi.py` prefill /
`rms_gemv_rope_multi.py` decode), but the RoPE launch itself is a standalone
row-wise rotation — bit-identical to this harness. The LUT is generated by
`llama32_1b_weights.py:generate_rope_lut` (the same half-split `[cos…, sin…]`
layout this harness uses). **The two other RoPE examples in the tree
(`rope_lut/`, `rope_sincos/`) compute the *interleaved* even/odd variant and do
not match llama — they are decoys; this entry measures the half-split kernel.**

## Numerical datapath (what "BF16 half-split RoPE" means here)

```
x,lut bf16 → (per row, per i<half):
  out[i]      = x[i]*cos[i] - x[i+half]*sin[i]   (bf16 mul, mul, sub)
  out[i+half] = x[i]*sin[i] + x[i+half]*cos[i]   (bf16 mul, mul, add)
→ bf16
```

- **Pure per-element rotation, no accumulation.** Each output element is a
  2-term combination of two bf16 inputs and two bf16 LUT values: two bf16
  multiplies and one bf16 add/sub. There is no reduction and no running
  accumulator, so — like Element-wise Add and unlike GEMM/RMSNorm/FlashAttention
  — there is **nothing to amplify rounding error** beyond the few bf16 ops per
  element.
- **The cos/sin are exact-bf16 LUT entries** (precomputed on host in f32, cast to
  bf16). The only quantization is the bf16 rounding of the products and the
  final sum. This is the same arithmetic a GPU/HF half-split RoPE does on bf16
  tensors.

The net effect is a measured `mean_rel_L1 ≈ 2.8e-3` — among the cleanest kernels
in the registry: above Element-wise Add (1.9e-3, a single rounding) because RoPE
does a handful of bf16 mul/add per element, but well below GEMM (~9.3e-3),
RMSNorm (~4.2e-3) and SiLU-and-Mul (~1.0e-2).

## Numerical accuracy

Verified element-wise over the full output against an FP32 reference (`x, lut`
upcast to f32, half-split rotate, cast to bf16):

| Metric (rows = 65536 = 32·2048, head_dim = 64, randn input) | Measured |
|---|---|
| `mean_rel_L1 = mean|out−ref| / mean|ref|` | **2.8e-3** |
| `rel_err max` | 7.8e-3 |
| `abs_err max` | 3.1e-2 |

- **`mean_rel_L1 = 2.8e-3`** — near the cleanest tier. A few bf16 mul/add per
  element, no accumulation to amplify error.
- **`rel_err max = 7.8e-3`** is small (RoPE is a rotation: `|out| ≈ |x|`, so no
  near-zero-reference blowup), so even the worst relative element is well within
  `rtol`.
- **`abs_err max = 3.1e-2`** is one bf16 ULP at the largest-magnitude outputs —
  covered by `atol = 5e-2`.
- **Accuracy is identical across `herd_x`/`herd_y` and across `rows`** (bit-for-bit
  `2.8e-3` at herd configs `1×1 … 8×1` and `2×4`, and rows ∈ {2048…65536}) — set
  only by the datatype, not the tiling or size. Decode-scale shapes (rows 8/32)
  measure slightly lower (2.4e-3/2.7e-3) only because the small position indices
  give smaller rotation angles, not because of any config dependence.
- **No reduction** → there is no accumulator and therefore no f32-vs-bf16
  accumulation question (the RMSNorm bf16-accumulation trap does not apply here).

## Parameters & constraints

RoPE is **memory-bound** (it streams `input`, `lut` in and `output` out for an
O(rows·head_dim) elementwise rotation; ~1 flop/byte). The herd is 2-D
(`sizes=[herd_x, herd_y]`), but a hardware limit pins the usable config:

| Knob | Value | Constraint → source |
|---|---|---|
| `herd_x` | **8** | AIE columns (≤ 8); `rows % (herd_x · herd_y) == 0` |
| `herd_y` | **1** | each tile uses 3 independent L3↔L1 DMAs (input in, lut in, output out); > 8 total tiles exhausts the shim DMA channels (`aircc`: *"no ShimNOCTile has sufficient DMA capacity"*). `2×4` (8 tiles) places; `8×2`/`4×4` (16 tiles) do not |
| `head_dim` | 64 | `head_dim % 16 == 0` (kernel vector width N=16); llama-3.2-1B head_dim |
| (DMA granularity) | 1 row/DMA | each `head_dim` row is one DMA (matching `rope_halfsplit.cc`'s single-row signature and the llama builders) — see Tunable space |

**The kernel cannot use the full 32-tile array** — the 3-DMA-per-tile shim-channel
demand caps it at **8 tiles** (one 8-column row, `herd_y = 1`; or the equivalent
`2×4`). Accuracy is independent of all of these.

> **`Makefile` default**: `ROWS=65536, HEAD_DIM=64, HERD_X=8, HERD_Y=1` (NPU2 bf16)
> — the best config, the llama-3.2-1B prefill RoPE-Q scale.

## Tolerances & reference

Element-wise over the **full output** against an FP32 reference: every element
must pass `|a−e| ≤ atol + rtol·|e|`.

| Output dtype | rtol | atol |
|---|---|---|
| bf16 | 1.6e-2 | 5e-2 |

- **Reference** = CPU FP32 half-split rotate (`x, lut` upcast to f32,
  `out[i]=x[i]·cos[i]−x[i+half]·sin[i]`, `out[i+half]=x[i]·sin[i]+x[i+half]·cos[i]`,
  cast to bf16) — the bf16-rounded reference a GPU/HF RoPE op is verified against.
  Inputs are `randn` (seed 0); the LUT is the half-split `[cos…, sin…]` layout
  with `theta = 500000` (llama-3.2 rope_base).
- **Matches the GPU op exactly.** HuggingFace `transformers`
  `models/llama/modeling_llama.py` computes
  `q_embed = q·cos + rotate_half(q)·sin` with `rotate_half(x) = cat(−x2, x1)`
  (`x1` = first half, `x2` = second half) — algebraically identical to the kernel's
  `out[i]=x[i]·cos−x[i+half]·sin`, `out[i+half]=x[i]·sin+x[i+half]·cos`. The
  harness reference reproduces this to within bf16 rounding (verified `max diff vs
  HF apply_rotary_pos_emb ≈ 6e-3`, i.e. one bf16 round).
- RoPE has **no dedicated PyTorch/HF unit test** with a published rtol/atol — it
  is a fused elementwise op validated through model integration. Per the registry
  convention (cite the framework's nearest general bf16 standard), `rtol = 1.6e-2`
  is PyTorch/vLLM's canonical bf16 tolerance; `atol = 5e-2` covers the worst-case
  single-element bf16 output rounding (`abs_err max ≈ 3.1e-2`, one ULP). With no
  accumulation, `mean_rel_L1 = 2.8e-3` sits far inside `rtol`. **The
  `rtol=0.2/atol=0.5/min_correlation=0.99` gate in the bf16 multi-launch builders'
  `__main__` test code is a stacked-error fused gate using the forbidden
  correlation metric — it is *not* used here.**

## Tested shapes

Shapes verified on NPU2 (bf16). **Best config is `herd_x=8, herd_y=1` for every
shape** (the shim-DMA-channel limit caps the herd at 8 tiles — see
[Parameters & constraints](#parameters--constraints)). `rows = 65536 = 32·2048`
is the llama-3.2-1B prefill RoPE-Q scale (32 query heads × 2048 seq); `16384 =
8·2048` is prefill RoPE-K (8 KV heads). Throughput is bandwidth (memory-bound).
`mean_rel_L1` is vs an FP32 reference.

| rows | head_dim | best config (hx/hy) | latency | bandwidth | mean_rel_L1 | abs_err max | Used by | Status |
|---|---|---|---|---|---|---|---|---|
| 8 | 64 | 8/1 | 83 µs | 0.04 GB/s | 2.4e-3 | 1.6e-2 | decode RoPE-K (8 KV heads) | ✅ |
| 32 | 64 | 8/1 | 82 µs | 0.15 GB/s | 2.7e-3 | 1.6e-2 | decode RoPE-Q (32 heads) | ✅ |
| 2048 | 64 | 8/1 | 105 µs | 7.5 GB/s | 2.8e-3 | 3.1e-2 | coverage | ✅ |
| 4096 | 64 | 8/1 | 118 µs | 13.3 GB/s | 2.8e-3 | 3.1e-2 | coverage / Qwen2.5-0.5B prefill RoPE-K (2 KV heads·2048) | ✅ |
| 16384 | 64 | 8/1 | 210 µs | 30.0 GB/s | 2.8e-3 | 3.1e-2 | prefill RoPE-K (8 KV heads) | ✅ |
| 65536 | 64 | 8/1 | 579 µs | 43.4 GB/s | 2.8e-3 | 3.1e-2 | prefill RoPE-Q (32 heads) | ✅ |
| 16384 | 128 | 8/1 | 285 µs | 44.2 GB/s | 2.8e-3 | 3.1e-2 | Qwen3-0.6B + Qwen3-1.7B prefill RoPE-K (8 KV heads, head_dim=128) | ✅ |
| 32768 | 128 | 8/1 | 477 µs | 52.8 GB/s | 2.8e-3 | 3.1e-2 | Qwen3-0.6B + Qwen3-1.7B + Qwen2.5-3B prefill RoPE-Q (16 heads, head_dim=128) | ✅ |
| 49152 | 128 | 8/1 | 667 µs | **56.6 GB/s** | 2.8e-3 | 3.1e-2 | Llama-3.2-3B prefill RoPE-Q (24 heads·2048, head_dim=128) | ✅ |
| 28672 | 64 | 8/1 | 303 µs | 36.4 GB/s | 2.8e-3 | 3.1e-2 | Qwen2.5-0.5B prefill RoPE-Q (14 heads·2048, head_dim=64) | ✅ |
| 24576 | 128 | 8/1 | 380 µs | 49.7 GB/s | 2.8e-3 | 3.1e-2 | Qwen2.5-1.5B prefill RoPE-Q (12 heads·2048, head_dim=128) | ✅ |
| 4096 | 128 | 8/1 | 149 µs | 21.1 GB/s | 2.8e-3 | 3.1e-2 | Qwen2.5-1.5B + Qwen2.5-3B prefill RoPE-K (2 KV heads·2048, head_dim=128) | ✅ |

> **Qwen2.5-1.5B** uses `head_dim = 128` (12 q-heads / 2 kv-heads): `rows = 12·2048 = 24576` (Q, new) and `2·2048 = 4096` (K, new at head_dim=128). Same half-split kernel, verified PASS at 2.8e-3.

> **Qwen2.5-0.5B** uses `head_dim = 64` (back to llama's dim, unlike Qwen3's 128): 14 q-heads / 2 kv-heads, so `rows = 14·2048 = 28672` (Q, new) and `2·2048 = 4096` (K, already in registry). Verified PASS at 2.8e-3.

> **`head_dim = 128`** (the two rows above) is the first registry coverage beyond llama's `head_dim = 64` — Qwen3-0.6B uses 16 q-heads / 8 kv-heads of `head_dim = 128`, so `rows = 16·2048 = 32768` (Q) and `8·2048 = 16384` (K). Same half-split `rope_halfsplit.cc` kernel, verified PASS at 2.8e-3 (accuracy set by the datapath, unchanged by head dim; `head_dim % 16 == 0` holds for 128).

> Bandwidth rises with rows (0.04 → 43.4 GB/s): the small shapes are
> **latency-dominated** by fixed launch overhead (~80 µs floor), while the large
> prefill shapes approach the DMA streaming rate. All shapes use the same best
> config (8×1); accuracy is bit-identical across shapes and herd configs (decode
> rows 8/32 read slightly lower from smaller rotation angles, not config).

**Reading the table**:
- **Memory-bound**: latency is gated by DMA. The kernel moves `3·rows·head_dim·2`
  bytes (input + lut in, output out, bf16); at 65536 rows that is 25.2 MB in
  579 µs ≈ 43.4 GB/s. Throughput is bandwidth, not GFLOP/s.
- **Accuracy** `mean_rel_L1 = 2.8e-3` — near the cleanest tier; set by the few
  bf16 ops per element, not the tile config.

## Tunable space & performance

The full tunable space is `(herd_x ≤ 8, herd_y ≤ 4)`, but the shim-DMA-channel
limit collapses it to a single best config:

**1. `herd_y > 1` cannot place past 8 tiles** — the 3-DMA-per-tile shim-channel
demand exceeds the hardware limit (the same root cause as Element-wise Add and
SiLU-and-Mul). The herd is capped at 8 tiles. Sweep at rows=65536:

| herd | tiles | result |
|---|---|---|
| 8×1 | 8 | ✅ **43.4 GB/s** (best) |
| 2×4 | 8 | ✅ 44.3 GB/s (≈ 8×1) |
| 8×2 | 16 | ❌ no ShimNOCTile has sufficient DMA capacity |
| 4×4 | 16 | ❌ no ShimNOCTile has sufficient DMA capacity |

**2. Within 8 tiles, more columns scale near-linearly** (each tile streams its
rows through its own DMA). Sweep of `herd_x` at rows=65536 (`herd_y=1`):

| herd_x | tiles | latency | bandwidth | speedup vs herd_x=1 |
|---|---|---|---|---|
| 1 | 1 | 4280 µs | 5.9 GB/s | 1.0× |
| 2 | 2 | 2213 µs | 11.4 GB/s | 1.9× |
| 4 | 4 | 1072 µs | 23.5 GB/s | 4.0× |
| 8 | 8 | **579 µs** | **43.4 GB/s** | **7.4×** |

**3. DMA granularity (rows per DMA).** The harness streams **one `head_dim` row
per DMA** — matching `rope_halfsplit.cc`'s single-row signature and the llama
prefill/decode builders. A batched variant (multiple rows per L3↔L1 DMA with
per-row `memref.subview` kernel calls) was implemented and tested, but the AIR
dependency pass mis-schedules the per-row subview writes under one bulk output
DMA: at `rows_per_dma = 2` it fails to place (`row-offset` shim error) and at
`rows_per_dma = 4` ~half the rows come back zero/NaN (`mean_rel_L1 ≈ 1.0`). So the
faithful one-row-per-DMA structure is used (it is what llama runs). This is a
toolchain limitation, not a kernel-math choice; head_dim=64 rows are small enough
that the ~80 µs launch floor, not per-DMA overhead, dominates the small shapes.

So **`herd_x = 8, herd_y = 1` is the best config** for every shape — full chip
width (the most the shim DMA channels allow). Accuracy is bit-identical across all
configs.

**llama config vs sweep-best.** The llama prefill builder already defaults to
`rope_herd_x = 8` — i.e. the sweep-best — so there is nothing left on the table
for prefill. The decode builder defaults to `rope_herd_x = 1` (only 8–32 rows;
herd_x=8 is ~no faster at that scale — both are ~82 µs, launch-floor-bound — so
the decode default is fine).

## How to reproduce (correctness + performance)

`rope_halfsplit.py` (compile-and-run mode, the default) runs the **correctness**
check via `XRTRunner`: full-output element-wise compare against the FP32
half-split reference; prints `[precision] mean_rel_L1=… | rel_err max=… | abs_err
max=… | rtol=… atol=…` and `PASS!` / `failed.` Add `--perf-iters N` for latency
(10 warmup iters excluded, N timed iters averaged, kernel-only). Every
tested-shapes row reproduces by setting `ROWS` (all use the same best config
`herd_x=8, herd_y=1`):

```bash
cd programming_examples/rope_halfsplit

# correctness — main shape (rows=65536), best config; compiles + runs on NPU2
make run ROWS=65536 PEANO_INSTALL_DIR=$PEANO_INSTALL_DIR

# any tested shape — change ROWS (8 / 32 / 2048 / 4096 / 16384 / 65536)
make run ROWS=16384 PEANO_INSTALL_DIR=$PEANO_INSTALL_DIR

# performance (latency → bandwidth) — run the script directly with --perf-iters
mkdir -p build_peano && cd build_peano
python3 ../rope_halfsplit.py --rows 65536 --head-dim 64 \
  --herd-x 8 --herd-y 1 --perf-iters 30
# bandwidth = 3·rows·head_dim·2 bytes / latency  (input+lut in, output out, bf16)

# full sweep (herd_x ∈ {1,2,4,8}, herd_y probes, all shapes):
#   programming_examples/kernel_registry/details/internal_RoPE_bf16/scripts/rope_sweep.sh
```

Notes:
- `herd_y` must keep `herd_x · herd_y ≤ 8` (shim DMA channel limit); `herd_x ≤ 8`.
- If the NPU is shared with other jobs, serialize on-device runs (e.g. with
  `flock`) so timing measurements aren't perturbed.
