# Irene — Round 1 Review

**Change**: Lightweight loop unrolling to reduce O(N*body_size) IR explosion
**File**: `mlir/lib/Util/Dependency.cpp` (+329/-12 lines)

## Summary

The change introduces a two-function system (`cloneRegionLightweight` + `loopUnrollFullLightweight`) that selectively clones segment/herd ops during loop unrolling, keeping only channel-related ops in herd bodies. The detection heuristic (`containsSegment` walk) routes shim-level loops to the lightweight path while preserving standard unrolling for all other cases.

## Findings

### C1 — Critical: Incorrect op construction — `SegmentOp`/`HerdOp` created without proper operand partitioning

**Location**: `Dependency.cpp:1173-1181`, `1204-1212`, and corresponding code in `cloneRegionLightweight` at lines 1044-1053, 1073-1082

The ops are created by passing all operands as a single `ValueRange`:
```cpp
SmallVector<Value> mappedOperands;
for (Value operand : op.getOperands()) {
    mappedOperands.push_back(mapper.lookupOrDefault(operand));
}
auto clonedSegment = builder.create<air::SegmentOp>(
    segmentOp.getLoc(), segmentOp.getResultTypes(), mappedOperands,
    segmentOp->getAttrs());
```

`SegmentOp` and `HerdOp` use `AttrSizedOperandSegments`. Their operands are partitioned into distinct groups (async_dependencies, sizes, segment/herd_operands). The custom builders expect these as separate `ValueRange` arguments and use them to:
1. Set `operand_segment_sizes` attribute correctly
2. Create body block arguments (2 per size dim + 1 per kernel operand)
3. Call `ensureTerminator()`

Passing a flat operand list with separate attrs (which include the old `operand_segment_sizes`) is fragile. The old `operand_segment_sizes` attribute might reference different operand counts if the mapper changes the number of async dependencies.

**Impact**: Body block arguments may have wrong count/types, downstream passes (air-to-std, air-dependency) that rely on block argument layout would produce incorrect IR or crash.

### C2 — Critical: `scf::ForOp` body cloning inside herds creates duplicate terminator

**Location**: `Dependency.cpp:1006-1028`

When a `scf::ForOp` is encountered inside a herd body, the code creates a new `ForOp` and recursively clones its body:

```cpp
auto clonedFor = builder.create<scf::ForOp>(
    forOp.getLoc(), mappedOperands[0], mappedOperands[1],
    mappedOperands[2], ValueRange(mappedOperands).drop_front(3));

// ...
OpBuilder forBuilder(&clonedFor.getRegion().front(),
                     clonedFor.getRegion().front().begin());
cloneRegionLightweight(forOp.getRegion(), clonedFor.getRegion(),
                       mapper, forBuilder, true);
```

`scf::ForOp::build` creates a body block with a `scf::YieldOp` terminator (for all cases, with or without init args). Inside `cloneRegionLightweight`, `builder.setInsertionPointToEnd(destBlock)` places the insertion point after this yield. Then:

1. Channel ops from the source body are cloned **after** the existing yield — invalid MLIR (ops after terminator)
2. The source's `scf::YieldOp` is also cloned (it's in the allowlist at line 987-989), creating a **duplicate** yield

This would produce:
```
scf.for %iv = ... {
    scf.yield  // auto-generated by ForOp::build
    air.channel.put ...  // cloned AFTER yield - invalid
    scf.yield  // duplicate from source - invalid
}
```

**Fix**: Erase the auto-generated yield before cloning:
```cpp
auto clonedFor = builder.create<scf::ForOp>(...);
// Erase the auto-generated yield
clonedFor.getBody()->getTerminator()->erase();
// Now clone body
cloneRegionLightweight(forOp.getRegion(), clonedFor.getRegion(),
                       mapper, forBuilder, true);
```

### S1 — Suggestion: Code duplication between `cloneRegionLightweight` and `loopUnrollFullLightweight`

**Location**: `Dependency.cpp:1044-1105` vs `1172-1234`

Both functions contain nearly identical code for:
- Creating a cloned SegmentOp with mapped operands
- Mapping results
- Mapping block arguments
- Calling `cloneRegionLightweight` recursively

And the same pattern for HerdOp. This is ~60 lines of duplication. Consider extracting a helper:

```cpp
static Operation *cloneSegmentOrHerdLightweight(
    Operation *op, IRMapping &mapper, OpBuilder &builder);
```

### S2 — Suggestion: `annotateFn` is called with corrupted builder

**Location**: `Dependency.cpp:1201-1203`, `1231-1233`

```cpp
if (annotateFn) {
    annotateFn(i, clonedSegment, builder);
}
```

After `cloneRegionLightweight` modifies the builder's insertion point (see Merlin's C2), `annotateFn` receives a builder pointing inside the segment body. If the annotation function creates ops (which annotation functions typically do), they'd be placed inside the segment instead of at the parent level.

**Location**: `Dependency.cpp:1117-1138`

auto tripCount = air::getStaticScfForTripCountAsInt(forOp);
// ... then later:

If `getStaticScfForTripCountAsInt` succeeds, the loop bounds are already static. The second check is redundant.
